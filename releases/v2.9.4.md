# WP Manager Pro v2.9.4

**Released:** 2026-03-18

## Summary

Patch release delivering a deep root-cause fix for the WordPress Media Library modal never opening across all media picker buttons in the plugin.

---

## Bug Fixes

### 🎯 Media Library modal — deep root cause resolved

**Symptom:** Clicking any "Media" or "Select" button (Agency Tools → Login Page logo/background, Coming Soon logo/background, Settings → Branding logo) did nothing, or showed a `prompt()` fallback. The `window.wmpOpenMedia` bridge existed and `window.wp.media` appeared to be available, yet the media frame threw `TypeError: Cannot read properties of undefined (reading 'View')` internally.

**Root cause:** Rollup's variable minifier happened to name the Lucide `StickyNote` icon component `const wp` at the **top level** of the built `index.js`. In a non-module browser script, a top-level `const` does **not** create a `window.wp` property — so `window.wp` remained intact — but it **does** shadow `window.wp` in the global scope for any code running in the same script environment. When `media-views.min.js` constructor functions (defined before the bundle loaded, but called after) resolved the unqualified identifier `wp`, they received the Lucide component object instead of WordPress's media namespace, making `wp.Backbone.View` undefined and aborting the modal.

**Fix:** Switched the Vite build to `format: 'iife'` in `vite.config.ts`. The entire bundle is now wrapped in an immediately-invoked function expression — all `const`, `let`, and `var` declarations inside the bundle are function-scoped and cannot leak into or shadow the global scope.

### 🛡️ `wmpOpenMedia` bridge hardened

The PHP-injected bridge function now captures `var _wp = window.wp` at the start of its body instead of relying on the unqualified `wp` identifier. This provides a belt-and-suspenders defence: even if a future build or third-party script introduces another `wp` shadow, the bridge will always resolve to the correct WordPress media object.
